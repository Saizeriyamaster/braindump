# クラウドとは何か

> 要点：「自社サーバーを持つ」から「必要な分だけ借りる」への変化。SaaSは完成品、PaaSは厨房、IaaSは土地を借りる

---

## オンプレミス vs クラウドの違い

| | オンプレミス | クラウド |
|-|------------|---------|
| 意味 | 自社でサーバーを購入・運用 | データセンターのリソースをネット越しに借りる |
| 初期コスト | 高い（ハード購入） | 低い（従量課金） |
| スケーリング | 難しい（買い足す必要） | 即時（ボタン一つで台数増やせる） |
| 管理負担 | ハードからOSまで自分で | クラウド側が大部分を管理 |
| 使い所 | 規制が厳しい業界、大企業 | スタートアップ〜中規模 |

---

## SaaS / PaaS / IaaS

レストランの例えで考える：

| モデル | 例え | 自分がやること | 例 |
|--------|------|--------------|-----|
| **IaaS** | 土地と厨房設備を借りる | OS〜アプリまで全部自分で | AWS EC2, GCP Compute Engine |
| **PaaS** | 厨房ごと借りる（道具あり） | アプリのコードだけ書く | Heroku, Cloud Run, App Engine |
| **SaaS** | 完成した料理を買う | 設定するだけ | Slack, Notion, Gmail, Salesforce |

```
       管理する範囲（多い ←→ 少ない）
IaaS ▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░
PaaS ▓▓▓▓░░░░░░░░░░░░░░░░
SaaS ▓░░░░░░░░░░░░░░░░░░░

▓ = 自分が管理  ░ = クラウドが管理
```

## 3大クラウドの比較

| | AWS | GCP | Azure |
|-|-----|-----|-------|
| 運営 | Amazon | Google | Microsoft |
| シェア | 最大（約32%） | 3位（約12%） | 2位（約22%） |
| 強み | サービスが最も豊富 | データ分析・AI/ML | Microsoftエコシステムとの統合 |
| AI/ML | SageMaker, Bedrock | Vertex AI, BigQuery ML | Azure OpenAI Service |
| 使い所 | とりあえずAWS | AI/ML・データ分析中心 | 企業のOffice 365環境と統合 |

## リージョンとアベイラビリティゾーン（AZ）

- **リージョン**：データセンターの地理的な場所（東京リージョン、バージニア北部リージョン等）
  - データの法的管轄、レイテンシに影響する
- **AZ（Availability Zone）**：リージョン内の独立した施設群。複数AZに分散してサービスを落とさない設計
- **マルチリージョン**：複数の地域にまたがって冗長化。大規模サービスや災害対策

## スケーリングの概念

| 種類 | 内容 | 例 |
|------|------|-----|
| **スケールアップ（垂直）** | 1台のスペックを上げる（CPU増、メモリ増） | t3.medium → t3.xlarge |
| **スケールアウト（水平）** | 台数を増やす | サーバー1台 → 10台 |
| **オートスケーリング** | アクセスに応じて自動で台数調整 | 夜は2台、昼のピークは20台 |

クラウドの真価はオートスケーリング。オンプレミスでは不可能。

---

## サーバーレス

「サーバーを意識しなくてよい」実行モデル。コードだけ書いて、実行基盤はクラウドに任せる。

- **AWS Lambda / GCP Cloud Functions**：関数単位で動く。アクセスがあった時だけ起動して課金される
- **使い所**：画像変換、通知送信、バッチ処理、APIのバックエンドなど
- **コールドスタート**：しばらく使われていない関数は起動に時間がかかる（数百ms〜数秒）

---

## コンテナとKubernetes（概要）

- **コンテナ（Docker）**：アプリとその依存関係を一つにパッケージングする技術。「どの環境でも同じように動く」を実現
- **Kubernetes（K8s）**：多数のコンテナを管理・スケールする仕組み（オーケストレーション）
- 「Dockerでコンテナ化 → K8sで本番運用」がモダンなデプロイの流れ

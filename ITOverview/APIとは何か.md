# APIとは何か

> 要点：レストランの注文口の例え。メニュー（仕様）通りに注文（リクエスト）すると料理（データ）が出てくる

---

## APIとは

**API（Application Programming Interface）**：ソフトウェア同士がデータをやり取りするための「窓口」。

レストランの例え：
- **客（クライアント）**：データを欲しいアプリ
- **注文口（API）**：決まった形式でリクエストを受け付ける窓口
- **厨房（サーバー）**：実際に処理する側
- **メニュー（API仕様書）**：何をどう頼めるかが書いてあるドキュメント

---

## REST API

現在最も普及しているAPI設計の流儀（アーキテクチャスタイル）。

### 基本ルール
- URLはリソース（データの種類）を表す
- HTTPメソッド（GET/POST/PUT/DELETE）で操作を表す
- データはJSONでやり取り

```
GET    /users          → ユーザー一覧を取得
GET    /users/123      → ID=123のユーザーを取得
POST   /users          → ユーザーを新規作成
PUT    /users/123      → ID=123のユーザーを更新
DELETE /users/123      → ID=123のユーザーを削除
```

### リクエスト・レスポンスの例

```http
# リクエスト
POST /api/predict
Content-Type: application/json

{
  "text": "この商品は最高です",
  "model": "sentiment-v2"
}

# レスポンス
HTTP/1.1 200 OK
Content-Type: application/json

{
  "label": "positive",
  "score": 0.97
}
```

## GraphQL

RESTの代替として使われる別のAPI規格。Facebookが開発。

| | REST | GraphQL |
|-|------|---------|
| エンドポイント | リソースごとに複数 | 1つだけ（`/graphql`） |
| データ取得 | サーバー側が決めた形で返す | クライアントが欲しい項目だけ指定できる |
| Over-fetching | 起きやすい（不要な項目も来る） | 起きない |
| 使い所 | 一般的なWebアプリ | フロントが複雑で柔軟なデータ取得が必要なとき |

---

## WebSocket

通常のHTTPは「リクエスト→レスポンス」の1往復で終わり。**WebSocket**は接続を維持して双方向通信ができる。

- チャットアプリ、リアルタイム通知、株価の更新など
- AIのストリーミングレスポンス（ChatGPTが少しずつ文字が出てくるやつ）もこれかSSE

---

## APIキーと認証

外部サービスのAPIを使うときはAPIキー（秘密の文字列）で認証する。

```python
import openai
client = openai.OpenAI(api_key="sk-...")  # これがAPIキー
```

**よくある認証方式**

| 方式 | 仕組み |
|------|--------|
| **APIキー** | 秘密の文字列をヘッダーや引数で渡す |
| **JWT** | 認証情報を暗号化したトークン |
| **OAuth 2.0** | 「Googleでログイン」のような第三者認証 |

---

## OpenAPI（Swagger）

REST APIの仕様を標準フォーマットで記述するための規格。

- `swagger.yaml` や `/docs`（FastAPIなら自動生成）にアクセスすると、どんなエンドポイントがあるか・どんなデータを送るか一覧で見られる
- クライアントコードの自動生成にも使われる

---

## Webhook

「イベントが起きたら通知してください」という逆向きのAPI。

- 通常のAPI：こちらから問い合わせる（ポーリング）
- Webhook：向こうからイベント発生時に叩いてもらう（プッシュ）

例：GitHubにpushされたら自動でデプロイ（CI/CD）、Stripeで決済完了したら注文確定メール送信

# ソフトウェアが動く仕組み

> 要点：スマホアプリもWebサービスも、結局は「CPUが命令を処理してメモリに置いてストレージに保存する」だけ

---

## ハードウェアの3点セット

| 部品 | 役割 | 例え |
|------|------|------|
| **CPU** | 命令を実行する演算装置 | 料理人 |
| **メモリ（RAM）** | 作業中のデータを一時置き | 調理台 |
| **ストレージ（SSD/HDD）** | データを永続保存 | 冷蔵庫・棚 |

- CPUはメモリ上のデータしか処理できない → ストレージから読み込んでメモリに乗せる作業が必要
- メモリは高速だが電源を切ると消える。ストレージは遅いが永続する
- GPUはCPUの並列版。行列演算を大量に同時処理できるのでDL学習に使う（知ってるはず）

---

## ソフトウェアの階層

```
アプリケーション（ブラウザ、Pythonスクリプト、スマホアプリ）
        ↓ システムコール
OS（Windows / macOS / Linux）
        ↓ ドライバ
ハードウェア（CPU / RAM / SSD / GPU）
```

**OS（オペレーティングシステム）の仕事**
- ハードウェアの抽象化：アプリが「ファイルを読む」と言ったら、実際のドライブアクセスはOSがやる
- プロセス管理：複数のアプリを並行して動かす（タイムシェアリング）
- メモリ管理：どのアプリがどのメモリ領域を使うか管理

---

## プロセスとスレッド

- **プロセス**：実行中のプログラム1つ。独立したメモリ空間を持つ
- **スレッド**：プロセス内の並列実行単位。同じメモリを共有
- Pythonは**GIL（Global Interpreter Lock）**のせいでスレッドが実質シングルコア → 並列処理には`multiprocessing`を使う理由

---

## なぜこれを知るか

- 「メモリが足りない」「CPUボトルネック」「I/Oバウンド」という会話がわかる
- モデルのバッチサイズ調整、GPUメモリ管理の理屈に直結
- 「Pythonは遅い」「なぜC++で書くのか」の理由がわかる（コンパイル言語とインタープリタ言語の違い）

---

## コンパイル言語 vs インタープリタ言語

| 種別 | 仕組み | 例 |
|------|--------|-----|
| コンパイル | 事前に機械語に変換して実行 → 速い | C, C++, Go, Rust |
| インタープリタ | 実行時に1行ずつ解釈 → 遅いが書きやすい | Python, JavaScript |
| 中間方式 | バイトコードにコンパイル後JIT実行 | Java, C# |

Pythonが遅い理由はこれ。ただしNumPy/PyTorchはC/C++実装なので数値計算は速い。

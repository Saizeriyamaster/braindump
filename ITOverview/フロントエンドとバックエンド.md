# フロントエンドとバックエンド

> 要点：フロントは「見た目と操作」、バックは「データ処理とロジック」。エンジニアの専門が分かれる理由もここにある

---

## 役割の分担

| | フロントエンド | バックエンド |
|--|--------------|------------|
| 場所 | ブラウザ（クライアント側）で動く | サーバー側で動く |
| 担う仕事 | UI表示、ユーザー操作の受付、APIの呼び出し | ビジネスロジック、DBアクセス、認証、計算処理 |
| 主な言語 | JavaScript / TypeScript | Python / Node.js / Go / Java / Ruby |
| 主なFW | React, Next.js, Vue, Svelte | FastAPI, Django, Express, Spring |

---

## フロントエンドの詳細

### React と Next.js の関係
- **React**：UIを「コンポーネント（部品）」単位で作るJSライブラリ
- **Next.js**：Reactをベースに、SSR/SSG/ルーティングなどを追加したフレームワーク
- 実務ではNext.jsが標準的。「Next.jsで書いてます」はReactも使っているということ

### State（状態）管理
- フロントエンドで「今何が選ばれているか」「データの取得状態」などを管理するのがState
- `useState`（React組み込み）や Zustand, Redux などのライブラリが使われる

### TypeScript
- JavaScriptに「型」を追加した言語。`変数の型を宣言してバグを事前に防ぐ`
- 大規模開発では型なしJSは破綻するのでTypeScriptがデファクト

---

## バックエンドの詳細

### APIサーバーの役割
- フロントからのリクエストを受け取り、DBに問い合わせて結果を返す
- ビジネスロジック（「このユーザーはこの操作ができるか」など）をここで実装

### AIエンジニアに身近なバックエンド：FastAPI
```python
from fastapi import FastAPI
app = FastAPI()

@app.get("/predict")
def predict(text: str):
    result = model.predict(text)   # MLモデルを呼ぶ
    return {"result": result}
```
- Pythonで書けるので機械学習モデルとの統合が簡単
- 型ヒントから自動でAPIドキュメント（Swagger UI）を生成してくれる

### ORMとは
- **ORM（Object-Relational Mapping）**：SQLを直接書かずにPythonのオブジェクトでDBを操作する仕組み
- 例：`User.query.filter_by(name="田中").first()` → 裏でSQLが動いている
- SQLAlchemy, Prisma, Django ORM など

---

## フルスタックとBFF

- **フルスタックエンジニア**：フロントもバックも両方書ける人
- **BFF（Backend for Frontend）**：フロント専用のバックエンドを作るパターン。モバイル用・Web用それぞれにAPIを最適化できる
- **Next.js の API Routes**：Next.jsのプロジェクト内にバックエンドAPIを書ける機能。フロントとバックを同一リポジトリで管理

---

## モノリスとマイクロサービス

| 方式 | 説明 | 使い所 |
|------|------|--------|
| **モノリス** | 全機能を1つのアプリにまとめる | スタートアップ、小〜中規模 |
| **マイクロサービス** | 機能ごとに独立したサービスに分割 | 大規模、チームが大きい |

マイクロサービスは管理が複雑になるので、小規模ではモノリスが正解なことが多い。

# Webアプリの仕組み

> 要点：「ユーザーがボタンを押す → リクエストが飛ぶ → サーバーが処理 → 結果が返る」の基本ループ

---

## 基本の流れ

```
ブラウザ（クライアント）
    ↓ HTTPリクエスト（GET / POST / PUT / DELETE）
Webサーバー / API サーバー
    ↓ SQLクエリ
データベース（PostgreSQL, MySQL等）
    ↑ データ返却
Webサーバー
    ↑ HTTPレスポンス（JSON or HTML）
ブラウザ
```

---

## HTTPとは

**HyperText Transfer Protocol**：Webブラウザとサーバーがデータをやりとりするためのルール（プロトコル）

**主なHTTPメソッド**

| メソッド | 意味 | 例 |
|--------|------|-----|
| `GET` | データを取得 | ページを開く、一覧を表示 |
| `POST` | データを新規作成 | フォーム送信、ユーザー登録 |
| `PUT` / `PATCH` | データを更新 | プロフィール編集 |
| `DELETE` | データを削除 | 投稿の削除 |

**HTTPステータスコード**（覚えておくと会話できる）

| コード | 意味 |
|--------|------|
| 200 | 成功 |
| 201 | 作成成功 |
| 400 | バッドリクエスト（クライアント側の間違い） |
| 401 | 未認証（ログインしてください） |
| 403 | 権限なし（ログインしてても見れない） |
| 404 | リソースが存在しない |
| 500 | サーバー内部エラー |

---

## HTTPSとSSL/TLS

- **HTTPS** = HTTP + 暗号化。通信を第三者に盗聴されても内容がわからない
- **TLS**（旧SSL）が暗号化の仕組み。ブラウザのURLバーに鍵マークが出るやつ
- 今は本番環境でHTTPはほぼ使わない

---

## DNSとは

- ドメイン名（`example.com`）をIPアドレス（`192.0.2.1`）に変換する仕組み
- 電話帳のようなもの。URLを入力するとDNSに問い合わせてサーバーのIPアドレスを得る

---

## セッションとCookie、JWT

認証状態（ログイン済みかどうか）をどう維持するか：

| 方式 | 仕組み | 特徴 |
|------|--------|------|
| **Cookie + セッションID** | サーバーがセッション情報を保持し、IDだけCookieに保存 | 古典的。サーバーに状態を持つ |
| **JWT（JSON Web Token）** | 認証情報をトークンに暗号化してクライアントに渡す | サーバーレスでも使える。モダンなAPI向け |

---

## SSR vs CSR vs SSG

フロントエンドのレンダリング戦略（Next.jsの文脈でよく出る）

| 方式 | 仕組み | 使い所 |
|------|--------|--------|
| **SSR**（Server-Side Rendering） | サーバーでHTMLを生成してブラウザに返す | SEOが必要なページ、データが動的 |
| **CSR**（Client-Side Rendering） | ブラウザがJSでHTMLを生成 | 管理画面、ログイン後のアプリ |
| **SSG**（Static Site Generation） | ビルド時にHTMLを生成 | ブログ、ドキュメント。最速 |

# ホスティングとデプロイ

> 要点：コードを書いたらどこに「置く」のか。GitHubにpushすると自動で本番反映される仕組みを理解する

---

## デプロイとは

**デプロイ（Deploy）**：書いたコードを実際にユーザーが使えるサーバーに配置・反映させること

```
開発者がコードを書く
    ↓ git push
GitHubにコードが上がる
    ↓ CIが自動でテスト実行
テスト通過
    ↓ CDが自動でサーバーに反映
本番環境でユーザーが使えるようになる
```

---

## 主要ホスティングサービス

### フロントエンド向け

| サービス | 特徴 | 使い所 |
|---------|------|--------|
| **Vercel** | Next.js開発元、フロントに最適化。GitHubと連携してPR毎にプレビュー環境が自動生成される | Next.js / React のデプロイ先としてデファクト |
| **Netlify** | Vercelに近い。Gatsby等にも強い | 静的サイト、Jamstack |
| **GitHub Pages** | GitHubが提供する無料ホスティング | ドキュメントサイト、ポートフォリオ |
| **Cloudflare Pages** | CDN+ホスティング。速い | 静的サイト、グローバル配信 |

### バックエンド向け

| サービス | 特徴 | 使い所 |
|---------|------|--------|
| **Railway** | GitHubとの連携が簡単。Dockerも使える。Herokuの後継的存在 | PythonバックエンドAPI、DBも一緒に立てられる |
| **Render** | Railwayと似た使い勝手。無料枠あり（スリープあり） | スタートアップ、個人開発 |
| **Fly.io** | コンテナをグローバル分散デプロイ。低レイテンシ | マルチリージョン展開 |
| **Heroku** | 老舗PaaS。2022年に無料枠廃止。今はRailway/Renderに移行が多い | 従来の定番だったが現在は減少 |

## CI/CD とは

**CI（Continuous Integration）**：コードをマージするたびにテスト・ビルドを自動実行
**CD（Continuous Deployment/Delivery）**：テスト通過後に本番環境へ自動デプロイ

「GitHubにpushするだけで自動でテストして本番に反映される」がCI/CDの理想形。

### 主要ツール

| ツール | 説明 |
|--------|------|
| **GitHub Actions** | GitHubに統合されたCI/CDツール。`.github/workflows/`にYAMLを書くだけ |
| **CircleCI** | 老舗CIサービス。設定の柔軟性が高い |
| **GitLab CI** | GitLabに内蔵 |

```yaml
# .github/workflows/deploy.yml の例
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: pip install -r requirements.txt
      - run: pytest          # テスト実行
      - run: ./deploy.sh     # デプロイスクリプト
```

## 環境の種類

| 環境 | 目的 | アクセス |
|------|------|---------|
| **ローカル環境** | 開発者のPC上での動作確認 | 自分だけ |
| **開発環境（dev）** | チームでの動作確認 | チーム内 |
| **ステージング環境（staging）** | 本番と同じ設定でのリリース前確認 | 社内 |
| **本番環境（production/prod）** | 実際のユーザーが使う環境 | 全員 |

「ステージングでは動いてたのに本番で落ちた」は環境差異が原因のことが多い。

---

## 環境変数と `.env` ファイル

APIキーやDBのパスワードはコードに直書きしない。**環境変数**として外出しする。

```bash
# .env ファイル（Gitにコミットしない）
DATABASE_URL=postgresql://user:password@localhost/mydb
OPENAI_API_KEY=sk-...
```

```python
import os
db_url = os.environ["DATABASE_URL"]  # 環境変数から読む
```

- `.env` はGitに含めない（`.gitignore` に追加）
- 本番環境では、クラウドのSecrets ManagerやVercel/Railwayの環境変数設定画面に入力する

## Docker と開発環境

**Docker**：アプリを「コンテナ」にパッケージングして「どの環境でも同じように動く」を実現

```dockerfile
# Dockerfile の例
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
```

- `docker-compose` で複数コンテナ（アプリ+DB等）をまとめて起動できる
- 「ローカルでDockerで動かして、本番もDockerで動かす」が一貫性のある運用
- K8s（Kubernetes）はDockerコンテナを大規模に管理するオーケストレーションツール

## デプロイ戦略

| 戦略 | 仕組み | 特徴 |
|------|--------|------|
| **ローリングデプロイ** | サーバーを少しずつ新バージョンに入れ替え | ダウンタイムなし |
| **ブルーグリーンデプロイ** | 新環境（グリーン）を用意して一気に切り替え | 即時ロールバック可能 |
| **カナリアリリース** | 一部ユーザーだけに新バージョンを先行リリース | 問題が起きた時の影響を限定 |

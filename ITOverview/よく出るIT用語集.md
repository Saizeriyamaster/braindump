# よく出るIT用語集

> エンジニアの会話やSlackで飛び交う用語のリファレンス。都度引けるようにしておく

---

## 開発プロセス

| 用語 | 意味 |
|------|------|
| **デプロイ（Deploy）** | コードを本番環境に反映させること |
| **リリース** | デプロイとほぼ同義。ユーザーに機能を提供できる状態にすること |
| **マイグレーション（Migration）** | ①DBのスキーマ変更。「カラム追加のマイグレーションを実行する」②システムの移行（AWSからGCPへの移行等） |
| **スキーマ変更** | DBのテーブル構造（カラム追加・削除・型変更等）を変えること。後方互換性に注意が必要 |
| **リファクタリング（Refactoring）** | 動作を変えずにコードの内部構造を改善すること |
| **テックデット（Technical Debt）** | 技術的負債。「後でやれば良い」として溜まった設計・実装の問題 |
| **スプリント** | アジャイル開発の1〜2週間の開発サイクル単位 |
| **スクラム** | アジャイル開発の手法の一つ。スプリント・スプリントレビュー・レトロスペクティブ等のイベントからなる |
| **PR / MR（Pull Request / Merge Request）** | コードレビューを経てブランチをマージするための申請。GitHubではPR、GitLabではMR |
| **コードレビュー** | 他の人のコードを読んで品質・バグを確認するプロセス |
| **LGTM（Looks Good To Me）** | コードレビューでの「問題なし・承認」のサイン |
| **ホットフィックス（Hotfix）** | 本番障害への緊急修正対応 |

## 環境・インフラ

| 用語 | 意味 |
|------|------|
| **ローカル / dev / staging / prod** | 開発環境の段階。ローカル（PC）→ 開発 → ステージング → 本番 |
| **コンテナ（Container）** | アプリとその依存関係をひとまとめにしたパッケージ（主にDocker） |
| **イメージ（Docker Image）** | コンテナの設計図。Dockerfileから作る |
| **オーケストレーション** | 多数のコンテナを自動管理する仕組み。KubernetesやECSが担う |
| **ロードバランサー（LB）** | 複数サーバーへのトラフィックを分散する装置・サービス |
| **CDN（Content Delivery Network）** | 世界中の拠点にコンテンツを配置して高速配信する仕組み。CloudFront, Cloudflare等 |
| **レイテンシ（Latency）** | 処理の遅延時間。リクエストを送ってレスポンスが返るまでの時間 |
| **スループット** | 単位時間あたりに処理できるリクエスト数。TPS（Transactions Per Second）とも |
| **冗長化（Redundancy）** | 障害時でもサービスが止まらないよう、同じシステムを複数用意すること |
| **フェイルオーバー** | 障害時に自動でバックアップ系に切り替わること |
| **SLA（Service Level Agreement）** | 稼働率・応答時間等の品質保証契約。「99.9% SLA」は年間8.7時間以内のダウンタイムまで許容 |

## コーディング

| 用語 | 意味 |
|------|------|
| **バグ（Bug）** | プログラムの不具合・誤動作 |
| **デバッグ（Debug）** | バグを見つけて修正すること |
| **スタックトレース** | エラー発生時に「どこで何が起きたか」を示す呼び出し履歴 |
| **ログ（Log）** | システムの動作記録。問題調査の基本 |
| **例外（Exception）** | 予期しないエラー。`try / except` でハンドリングする |
| **ヌルポ（NullPointerException）** | 値がない（null/None）オブジェクトを使おうとしたときのエラー。頻出バグ |
| **非同期（Async）** | 処理の完了を待たずに次の処理を進める方式。`await`・`async` キーワード |
| **同期（Sync）** | 処理の完了を待ってから次に進む方式 |
| **レース条件（Race Condition）** | 複数の処理が同時に走り、実行順序によって結果が変わるバグ |
| **メモリリーク** | 使い終わったメモリが解放されず、徐々にメモリ消費が増えていく問題 |
| **ハードコーディング** | 値をコードに直書きすること（APIキーをコードに書くのもこれ）。避けるべきとされる |
| **マジックナンバー** | コードに意味なく数値が登場すること（`if count > 87`）。定数として名前をつけるべき |
| **リントエラー（Lint）** | 静的解析ツールが指摘するコードスタイルの問題 |

## アーキテクチャ・設計

| 用語 | 意味 |
|------|------|
| **モノリシック（Monolithic）** | 全機能が1つのアプリに統合された構成 |
| **マイクロサービス** | 機能ごとに独立したサービスに分割する構成 |
| **API ゲートウェイ** | 複数のバックエンドサービスへのリクエストを一元管理する玄関口 |
| **イベント駆動（Event-Driven）** | イベント（ユーザー操作・DBの変更等）が起きたら処理を実行する設計 |
| **メッセージキュー** | サービス間で非同期にメッセージを送受信する仕組み。AWS SQS, RabbitMQ等 |
| **Pub/Sub** | 発行者（Publisher）がメッセージを送ると、購読者（Subscriber）が受け取る仕組み |
| **CRUD** | Create / Read / Update / Delete。DBの基本操作の頭文字 |
| **DRY原則** | Don't Repeat Yourself。同じコードを繰り返し書かない原則 |
| **SOLID原則** | オブジェクト指向設計の5つの基本原則（Single responsibility, Open-closed等） |
| **MVP（Minimum Viable Product）** | 最小限の機能で動くプロダクト。「まず動くものを」の発想 |

## バージョン管理（Git）

| 用語 | 意味 |
|------|------|
| **コミット（Commit）** | コードの変更を記録・保存すること |
| **ブランチ（Branch）** | メインのコードから分岐した作業ライン |
| **マージ（Merge）** | ブランチの変更を別のブランチに取り込むこと |
| **コンフリクト（Conflict）** | 同じ箇所を別々に変更したときに発生する衝突。手動で解決が必要 |
| **リベース（Rebase）** | コミット履歴を整理してブランチを別のポイントに移動する操作 |
| **スタッシュ（Stash）** | コミットせずに変更を一時退避する機能 |
| **タグ（Tag）** | 特定のコミットに付ける名前。バージョン番号（v1.0.0等）によく使う |
| **main / master** | デフォルトブランチの名前。最近はmainが主流 |
| **フォーク（Fork）** | リポジトリをコピーして自分の管理下に置くこと。OSSへの貢献フロー |

## セキュリティ

| 用語 | 意味 |
|------|------|
| **認証（Authentication）** | 「あなたは誰か」を確認すること。ログイン |
| **認可（Authorization）** | 「あなたに何が許可されているか」を確認すること。権限チェック |
| **XSS（Cross-Site Scripting）** | 悪意あるスクリプトをWebサイトに埋め込む攻撃 |
| **CSRF（Cross-Site Request Forgery）** | 偽サイトから正規サービスへの不正リクエストを送らせる攻撃 |
| **SQLインジェクション** | 悪意ある文字列をSQLクエリに混ぜてDBを不正操作する攻撃 |
| **ハッシュ化** | パスワード等を一方向に変換して保存すること。bcrypt, SHA-256等 |
| **暗号化** | データを鍵で変換して第三者に読めなくすること。TLS, AES等 |
| **脆弱性（Vulnerability）** | セキュリティ上の弱点・穴 |
| **ペネトレーションテスト** | 意図的に攻撃してセキュリティ上の問題を発見するテスト |

| **ORM** | Object-Relational Mapping | SQLをオブジェクトで操作する仕組み |
| **PoC** | Proof of Concept | 概念実証。「本当に動くか試す」フェーズ |
| **KPI** | Key Performance Indicator | 重要業績評価指標（ビジネス側でも使う） |
| **SaaS / PaaS / IaaS** | as a Service 系 | [[クラウドとは何か]] 参照 |
| **CI/CD** | Continuous Integration/Deployment | 自動テスト・デプロイのパイプライン |
| **TDD** | Test-Driven Development | テストを先に書いてから実装する開発手法 |
| **OKR** | Objectives and Key Results | 目標と主要な成果の設定フレームワーク |
